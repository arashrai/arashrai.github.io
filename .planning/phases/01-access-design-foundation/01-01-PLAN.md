<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Guest browser -> localStorage | Tier value stored client-side; guest can inspect/modify via DevTools |
| Guest browser -> HTML source | All content is in the HTML regardless of gate state; View Source bypasses the gate |
| Guest input -> SHA-256 comparison | Password input is hashed and compared to stored hashes; hashes are visible in auth.js source |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | auth.js TIER_HASHES | accept | Password hashes are in client-side JS source. SHA-256 obscures casual viewing but a determined user could reverse common passwords. Accepted: this is a wedding "polite lock", not a security boundary. Content is not sensitive. |
| T-01-02 | Tampering | localStorage narsh-tier | accept | Guest could set localStorage to "full" via DevTools to bypass tier restriction. Accepted: wedding content tiers are a UX convenience, not access control. No sensitive data differs between tiers. |
| T-01-03 | Information Disclosure | HTML source content | accept | All page content is in the HTML source regardless of auth state. CSS/JS gate is UI-level only. Accepted: static site constraint (no server-side rendering). Wedding content is not confidential. |
| T-01-04 | Denial of Service | crypto.subtle.digest | accept | SHA-256 hashing is fast; no rate limiting needed. A user could spam the form but only affects their own browser. No server resources involved. |
| T-01-05 | Information Disclosure | Placeholder passwords in source | accept | The placeholder passwords "day2guest" and "fullaccess" (per D-13) are development-only values. Once deployed, their SHA-256 hashes are in auth.js source, meaning anyone who views the source could attempt to reverse them. IMPORTANT: Before production deploy, replace these with unique passwords that are not reused from any personal accounts. Do not reuse personal passwords as wedding site passwords -- they will be effectively public. |
</threat_model>
